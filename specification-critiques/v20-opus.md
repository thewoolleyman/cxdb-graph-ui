# CXDB Graph UI Spec — Critique v20 (opus)

**Critic:** opus (Claude Opus 4.6)
**Date:** 2026-02-24

## Prior Context

The v18 and v19 cycles have not been acknowledged or revised. V18 had two critics (opus and codex) raising 6 issues total: unreachable dead code in `fetchFirstTurn`, missing gap recovery holdout scenario, contradictory detail panel cross-instance ordering, `/api/dots` response format inconsistency, undefined node ID loading for inactive pipelines, and graph ID extraction failing for `graph`/quoted names. V19 had two critics (opus and codex) raising 5 issues total: merged `hasLifecycleResolution` suppressing error/stale heuristics in parallel branches, unbounded gap recovery pagination, `determineActiveRuns` failing on unreachable instances, pipeline liveness undefined during outages, and `/api/dots` format contradiction (re-raised). None of these have been addressed yet. This critique raises new issues.

## Issue #1: DOT node attribute parsing specification lacks handling for multi-line attribute values and string concatenation

### The problem

Section 3.2 specifies that the `/dots/{name}/nodes` endpoint parses node attributes from DOT source, supporting quoted (`key="value"`) and unquoted (`key=value`) values with escape sequences `\"`, `\n`, and `\\`. However, DOT files generated by Attractor pipelines commonly use multi-line prompt attributes. The DOT language supports string concatenation with `+`:

```dot
implement [shape=box, prompt="First line of prompt.\n" +
  "Second line continues here.\n" +
  "Third line with more detail."]
```

The spec's parsing rules do not mention string concatenation. An implementer writing a DOT parser based solely on Section 3.2 would treat the first `"` to the next unescaped `"` as the complete value, breaking at the first closing quote and losing the concatenated portions. The holdout scenario "DOT file with long prompt text" expects the full prompt to be available in the detail panel, but doesn't test whether the prompt was assembled from concatenated strings.

Additionally, the spec doesn't address how the server handles attribute values that span literal newlines within quotes (which DOT also allows). A naive line-by-line parser would fail on:

```dot
implement [prompt="line one
line two
line three"]
```

### Suggestion

Either (a) extend Section 3.2's parsing rules to explicitly support `+` string concatenation and multi-line quoted values, noting that the concatenated fragments should be joined without any separator (the `+` operator in DOT concatenates without inserting characters), or (b) add a constraint stating that Attractor-generated DOT files do not use string concatenation or multi-line values, limiting the parser's scope. Option (a) is safer given that the spec cannot control DOT file generation format.

## Issue #2: `resetPipelineState` clears per-context status maps but does not clear `knownMappings` — stale context entries persist and match the old run

### The problem

Section 6.1 step 3 calls `resetPipelineState(pipeline.graphId)` when the active `run_id` changes for a pipeline. The function "clears per-context status maps, `lastSeenTurnId` cursors, and per-pipeline turn cache for all contexts that belonged to the old run." However, `knownMappings` — the persistent context-to-pipeline cache from Section 5.5 — is not cleared.

After a reset, `knownMappings` still contains entries like `(0, 33) -> { graphName: "alpha_pipeline", runId: "OLD_RUN_ID" }`. On the next poll cycle, `determineActiveRuns` iterates `knownMappings` and finds these old-run contexts. They are grouped by `runId`, and the active run is selected by highest `created_at_unix_ms`. The old contexts are correctly excluded from the active run because their `run_id` differs from the new run's. So far, no bug.

But there is a subtlety: if CXDB garbage-collects old contexts (or they are manually deleted), then `lookupContext` will fail for these stale `knownMappings` entries because they no longer appear in the context list from step 1. The spec doesn't define what happens when `lookupContext` can't find a context that exists in `knownMappings` but not in the current poll's context list — this is the same class of issue as v19-opus Issue #3 but triggered by context deletion rather than instance unreachability.

More practically, if the old run had 50 contexts and the new run has 2, every poll cycle iterates all 52 `knownMappings` entries for this pipeline, calling `lookupContext` 52 times when only 2 will produce results for the active run. Over time, with many runs, this grows unboundedly.

### Suggestion

When `resetPipelineState` fires due to a run change, also remove `knownMappings` entries whose `runId` matches the old run. These entries are no longer useful — they will never match the active run, and if the same context IDs somehow appear in a future run, they would have different `RunStarted` data and should be re-discovered. Add a note that `knownMappings` retains entries for the new run and for non-matching (null) entries (since those are still valid negative caches).

## Issue #3: CSS status selectors target `polygon` and `ellipse` but miss `path` elements used by some Graphviz node shapes

### The problem

Section 6.3 defines CSS selectors for status coloring:

```css
.node-pending polygon, .node-pending ellipse   { fill: #e0e0e0; }
.node-running polygon, .node-running ellipse   { fill: #90caf9; ... }
```

These selectors target `polygon` and `ellipse` SVG elements. However, Graphviz's SVG output uses different SVG primitives for different DOT shapes:

- `box` → `polygon` (4-point rectangle)
- `diamond` → `polygon` (4-point diamond)
- `Mdiamond` → `polygon`
- `Msquare` → `polygon`
- `ellipse` (default shape) → `ellipse`
- `hexagon` → `polygon`
- `parallelogram` → `polygon`

This covers the shapes listed in Section 7.3. But Graphviz can also render shapes using `path` elements depending on the shape and Graphviz version. Specifically, `@hpcc-js/wasm-graphviz` (the WASM build) may render certain shapes differently from native Graphviz. If any shape renders as a `path` element (which is common for rounded shapes like `Mrecord` or when `style=rounded` is applied), the CSS fill rules will not match and the node will not be colored.

The holdout scenario "Nodes rendered with correct shapes" lists the expected shapes but doesn't verify that status coloring applies correctly to each shape type.

### Suggestion

Add a `path` selector to each CSS rule as a safety net:

```css
.node-pending polygon, .node-pending ellipse, .node-pending path { fill: #e0e0e0; }
```

Alternatively, use a more general approach: target the first child shape element of the node group, or use `[fill]` attribute selectors within `.node` groups. Also consider adding a holdout scenario that verifies status coloring is visible on each of the six node shapes listed in Section 7.3.
