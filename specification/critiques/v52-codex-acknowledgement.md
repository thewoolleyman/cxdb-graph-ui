# CXDB Graph UI Spec — Critique v52 (codex) Acknowledgement

Both issues were valid and applied. Issue #1 identified a real multi-instance ordering bug: the active run selection algorithm compared `context_id` values across CXDB instances, but `context_id` is a per-instance monotonic counter that starts at 1 on each server independently. An old run on a high-counter instance (e.g., context_ids 500–550) would always defeat a newer run on a low-counter instance (e.g., context_ids 12–20), causing the UI to display stale data forever when runs migrate between servers. The fix switches to lexicographic comparison of `run_id` ULIDs, which encode the wall-clock creation time in the first 48 bits and are generated by Kilroy (not CXDB), making them safely comparable across instances. Issue #2 identified a prose/pseudocode contradiction: the paragraph at Section 5.5 said kilroy-prefixed contexts from the supplemental list were "only appended when CQL returned empty (to avoid duplicates)", while the pseudocode correctly performs dedup-based merging regardless of CQL result count. The fix rewrites the prose to accurately describe the always-run dedup merge.

## Issue #1: Cross-instance `context_id` comparison picks the wrong active run

**Status: Applied to specification**

The critique is correct. CXDB's `context_id` is allocated by `TurnStore::next_context_id` (`turn_store/mod.rs` lines 347-348), which starts at 1 and increments independently on each server instance. The value is persisted across restarts (recovered via `update_counters()` on `TurnStore::open`) but is never synchronized across instances. The old `determineActiveRuns` pseudocode compared `max(parseInt(c.contextId, 10) FOR c IN contexts)` across run groups and selected the run with the highest such value. With two CXDB instances, this comparison crosses instance boundaries: CXDB-0 might have accumulated 550 contexts (old run) while CXDB-1 has only 20 (new run) — the algorithm would pick CXDB-0's old run every time.

The fix uses `run_id` ULID lexicographic comparison instead. Verified from the Kilroy source: `NewRunID()` in `internal/attractor/engine/runid.go` uses `ulid.New(ulid.Timestamp(t), entropy)` from `github.com/oklog/ulid/v2`. ULIDs encode a 48-bit millisecond timestamp in the most significant bits, so lexicographic string comparison is equivalent to time-ordered comparison. The `run_id` is generated by Kilroy at run start time (not by CXDB), so it is a globally comparable creation-time signal that does not depend on any per-instance counter.

Changes:
- `specification/cxdb-graph-ui-spec.md` Section 5.5 "Multiple runs" paragraph: Replaced `context_id`-based active run selection with `run_id` ULID lex order; added "Why not `context_id`" and "Why not `created_at_unix_ms`" sub-explanations; cited `internal/attractor/engine/runid.go`
- `specification/cxdb-graph-ui-spec.md` Section 6.1 step 3: Updated the active run determination description to reference `run_id` ULID lex max and explain why `context_id` cannot be used cross-instance
- `specification/cxdb-graph-ui-spec.md` `determineActiveRuns` pseudocode: Replaced `highestContextId`/`maxContextId` variables with ULID string comparison (`IF activeRunId IS null OR runId > activeRunId`); updated the comment block to explain the ULID ordering rationale and the cross-instance counter hazard
- `specification/cxdb-graph-ui-spec.md` post-pseudocode prose: Updated "CXDB context IDs are monotonically increasing integers allocated from a global counter" to accurately describe per-instance allocation

## Issue #2: Supplemental-merge prose still says "only append when CQL returned empty"

**Status: Applied to specification**

The critique is correct. The prose paragraph in Section 5.5 ("CQL discovery limitation") ended with: "`kilroy/`-prefixed contexts found in the supplemental list are only appended to `contexts` when CQL returned empty (to avoid duplicates)." This was the original pre-v51 behavior — v51-codex's fix updated the pseudocode to use a `cqlContextIds` dedup set and merge supplemental kilroy-prefixed contexts regardless of CQL result count, but the prose was not updated to match. An implementer reading only the prose paragraph (rather than the pseudocode) would implement the old broken behavior: skip supplemental merge when CQL returns any contexts, silently dropping active runs from mixed deployments.

The fix rewrites the sentence to accurately describe the dedup-based always-merge behavior: the supplemental fetch now serves three explicitly numbered roles (expanded from two), and the sentence about when kilroy-prefixed contexts are appended now states they are **always** merged using a `context_id` dedup set built from CQL results.

Changes:
- `specification/cxdb-graph-ui-spec.md` Section 5.5 "CQL discovery limitation" paragraph: Rewrote the supplemental fetch roles from two to three (explicitly calling out case (2) — mixed deployment active-run discovery); replaced "only appended when CQL returned empty" with "always merged into `contexts` using a dedup set built from `context_id` values already present in the CQL results"
